# NUNCHI KIOSK — FastAPI AI 서버 프로젝트 개요

## 프로젝트 정의
**NUNCHI (눈치) KIOSK**
LLM Agentic AI 기반 배리어프리 자율주문 키오스크

> 음성 AI와 MCP 프로토콜을 활용하여 AI 에이전트가 메뉴 탐색, 추천, 주문, 결제를 보조하거나 자율 제어하는 차세대 키오스크

사용자가 자연어 대화만으로 메뉴 탐색 → 추천 → 담기 → 결제까지 완료할 수 있는 대화형 주문 시스템을 목표로 한다.

---

## 시스템 아키텍처

```text
[Hardware Layer]       범용 스탠드형 키오스크 + 핀 마이크
[Presentation Layer]   React UI + AI 오버레이 + WebSocket UI 제어
[AI Agent Layer]       FastAPI — STT → LLM → MCP Tool Router → TTS + 세션 관리
[Backend Layer]        Spring Boot — 메뉴/주문/결제/세션/통계 API + PostgreSQL
[Infrastructure]       Docker + AWS + Nginx
```

FastAPI는 React와 Spring 사이에서 동작하는 AI 서버이며, OpenAI API와 MCP Tool을 사용해 음성 대화와 시스템 제어를 담당한다.

## FastAPI 서버의 역할

FastAPI는 다음을 담당한다.

1. **음성 처리** — 오디오 입력 수신, STT 변환, TTS 응답 생성
2. **대화 처리** — 사용자 의도 파악, 주문 대화 흐름 제어, 추천 응답 생성
3. **MCP Tool 실행** — 메뉴 조회, 주문, 결제, UI 제어 Tool 선택 및 실행
4. **Spring 연동** — 메뉴/주문/결제/세션 API를 호출해 실제 데이터와 상태를 반영
5. **React 연동** — UI 제어 이벤트, 추천 신호, 확인 모달 트리거 전달
6. **세션 관리** — 대화 세션, 주문 상태, Tool 호출 이력 관리

---

## MCP 도구 정의

| MCP Tool | 역할 |
|----------|------|
| `DB Tool` | 메뉴 조회, 판매 데이터 조회, 추천 근거 조회 |
| `UI Control Tool` | 키오스크 화면 제어, 메뉴 강조, 장바구니 반영, 확인 모달 표시 |
| `Payment Tool` | 주문 확정, 카드 결제, 정맥인증 결제 흐름 제어 |

FastAPI는 MCP Tool의 실제 실행 주체이며, 필요 시 Spring API와 React UI를 함께 제어한다.

## 주문 모드

### 일반 UI 모드
- 터치 기반 주문이 기본 흐름
- 필요할 때만 LLM 채팅 기능을 사용
- AI가 화면을 조작할 때는 사용자 확인 모달이 선행되어야 한다
- 추천 팝업은 닫을 수 있어야 하며, 터치 우선 원칙을 해치지 않아야 한다

### 아바타 UI 모드
- 캐릭터 아바타와 음성 대화로 주문 진행
- 메뉴 추천, 장바구니 담기, 주문 확인, 결제 유도까지 AI가 주도한다
- 첫 방문자도 자연스럽게 사용할 수 있는 대화 흐름이 필요하다

## 음성 AI 파이프라인

```text
1. 음성 수신     키오스크 마이크에서 오디오 입력 수신
2. STT 변환      Whisper로 음성 → 텍스트 변환
3. 의도 분류     LLM이 주문/추천/질문/결제 의도를 분류
4. 개체 추출     메뉴명, 수량, 옵션, 결제 의도 등 추출
5. Tool 실행     MCP Tool 선택 및 체이닝 실행
6. TTS 응답      처리 결과를 음성으로 합성해 사용자에게 전달
```

필요 시 각 단계에서 세션 상태를 참조하고, 이전 대화 문맥을 유지해야 한다.

## 눈치 기능

사용자가 도움을 명시적으로 요청하지 않아도, 망설임 신호를 감지해 먼저 추천을 제안한다.

감지 신호 예시:
- **체류 시간**: 특정 화면에 오래 머무름
- **반복 탐색**: 같은 메뉴나 카테고리를 반복 확인
- **침묵**: 응답 없이 일정 시간 정지
- **헤징 발화**: "음", "뭐가 좋지", "추천해줘" 같은 불확실 표현
- **음성 불확실성**: STT 신뢰도 낮음

FastAPI는 이런 신호를 받아 추천 트리거를 판단하고, 적절한 응답 또는 UI 액션으로 연결한다.

---

## 추천 로직 원칙

- 추천은 반드시 실제 데이터 기반이어야 한다
- 메뉴/가격/재고/판매량은 Spring API 또는 확정된 데이터 소스를 기준으로 판단한다
- 임의 추천보다는 "오늘 많이 팔린 메뉴", "카테고리 인기 메뉴", "비슷한 대안"처럼 근거를 제시한다
- 추천 결과는 대화 응답과 UI 강조를 함께 고려한다

예시:
- `get_top_menus`
- `get_menus_by_category`
- `get_menu_detail`
- `search_menu`

## React ↔ FastAPI ↔ Spring 통신 구조

```text
React
  ↕ REST / WebSocket
FastAPI
  ↕ HTTP REST
Spring Boot
  ↕
PostgreSQL
```

### 역할 분담
- **React**: UI 렌더링, 터치 이벤트, 음성 입력 수집, 눈치 관련 프론트 이벤트 수집
- **FastAPI**: 음성 파이프라인, LLM 응답, MCP Tool 실행, UI 제어 이벤트 생성
- **Spring**: 메뉴/주문/결제/세션/통계 데이터의 실제 소스

### 연동 원칙
- FastAPI는 Spring API 계약이 확정된 뒤 그 계약을 기준으로 구현한다
- UI 제어는 React와의 WebSocket 또는 유사한 실시간 채널을 우선 고려한다
- 민감한 결제 정보는 FastAPI에서 최소한만 다루고, 실제 처리는 백엔드 시스템 기준으로 위임한다

## 개발 원칙

1. **MCP 우선 설계**: 기능은 대화 응답이 아니라 Tool 실행 관점에서도 설계한다.
2. **실제 데이터 우선**: 메뉴, 가격, 주문 상태는 하드코딩하지 않고 실제 API 기준으로 처리한다.
3. **모듈 분리**: Router, Service, Adapter, MCP Tool, Domain 역할을 분리한다.
4. **계약 우선 구현**: Spring과 React의 인터페이스가 확정되지 않으면 추측 구현을 최소화한다.
5. **상태 일관성 유지**: 세션, 장바구니, 결제 흐름이 서로 어긋나지 않도록 관리한다.
6. **관측 가능성 확보**: Tool 호출, 외부 API 오류, 세션 흐름은 디버깅 가능해야 한다.
